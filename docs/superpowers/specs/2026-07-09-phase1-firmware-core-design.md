# Phase 1: Firmware Core + kline_sim.py — Design

**Status:** Approved by user 2026-07-09. Feeds into a `writing-plans` implementation plan.

**Scope:** PLAN.md §9 Phase 1 row — `kline_sim.py` + firmware core (KWP stack,
scheduler, logger). Gate: "Full simulated ride CSV on bench." User confirmed

## 1. The core architectural problem

`kline_kwp` (the protocol state machine) must run identically in three
contexts that don't share hardware:

1. PlatformIO's `native` test env — no ESP32, no serial port at all.
2. Real ESP32 hardware (the bare board you have) — UART2 on GPIO17/16.
3. Against `kline_sim.py` running on the PC, either as a native-test fixture
   source or live over a real serial port.

**Decision: a small transport abstraction**, not `HardwareSerial` calls
embedded directly in protocol logic. Considered and rejected: (a) skip native
mocking and validate only against real hardware — faster to a demo, but loses
fast hardware-independent regression testing, cutting against how Phase 0 was
built; (b) hand-written fixtures decoupled from `kline_sim.py`'s actual
response logic — simpler short-term, but the fixtures and the simulator drift
apart and native tests stop meaning anything.

```cpp
class KLineTransport {
 public:
  virtual ~KLineTransport() = default;
  virtual void write(const uint8_t* data, size_t len) = 0;
  // Returns bytes actually read (may be < len on timeout).
  virtual size_t read(uint8_t* buf, size_t len, uint32_t timeout_ms) = 0;
  virtual uint32_t now_ms() = 0;
};
```

- `Esp32UartTransport` (esp32dev env only, wraps `Serial2`) — real hardware.
- A native test double, fed either hand-authored byte sequences for
  edge-case tests (checksum errors, timeouts) or, for the end-to-end
  "full simulated ride" test, sequences generated by literally invoking
  `kline_sim.py`'s response-building functions (imported/ported logic, not
  re-typed) so the fixture and the live simulator can't drift apart.

## 2. Bench test topology (Phase 1, no L9637D)

Bare ESP32 UART2 (GPIO17 TX / GPIO16 RX, per `hardware/wiring.md`) wired
**directly** to a USB-TTL/FTDI adapter's RX/TX (crossed) + common GND. Pure
TTL-level protocol testing — no K-line transceiver, no 12V, no pull-up
resistor. `kline_sim.py` runs on the PC end of that USB-TTL adapter. This is
distinct from `hardware/build_guide.md` Stage C (which explicitly goes
through the real L9637D and is gated under Phase 2) — Phase 1's bench test is
a *protocol-logic* validation, Stage C is an *electrical* validation.

## 3. Storage abstraction (no SD card yet)

```cpp
class RideStorage {
 public:
  virtual ~RideStorage() = default;
  virtual bool open_for_write(const char* filename) = 0;
  virtual void write_line(const char* line) = 0;
  virtual void flush() = 0;
  virtual void close() = 0;
};
```

Phase 1: `LittleFsStorage` (ESP32 internal flash — no extra wiring, plenty of
headroom per Phase 0's build showing 20% flash used). Phase 2: `SdStorage`
implements the same interface once the SD card is wired (`hardware/wiring.md`
GPIO 5/18/19/23). `ride_logger` depends only on the interface.

## 4. `kline_sim.py`

`firmware/tools/kline_sim.py` — standalone script, NOT part of the
`pipeline/` uv project (different concern, different lifecycle). Uses a
[PEP 723](https://peps.python.org/pep-0723/) inline script-metadata block
declaring `pyserial` as its only dependency, so `uv run firmware/tools/kline_sim.py`
works with no separate venv setup.

Responsibilities:
- Answers fast-init (tried first by firmware) and 5-baud init (fallback).
- Answers `tester present` (Mode 0x3E-style keepalive) with the required ACK.
- Serves Mode 01 (current data) using the exact 19-signal schema from
  `CLAUDE.md`/`pipeline/src/kl/schema.py`/`firmware/src/pid_schema.h` —
  correctly **not** responding to PIDs tagged "Likely unsupported" (PEDAL_D,
  PEDAL_E, CMD_THROTTLE, CTRL_MODULE_V), so firmware's handling of an
  unsupported-PID response gets exercised for real, not just assumed.
- Serves Mode 02 (freeze frame), Mode 03 (stored DTCs), Mode 07 (pending),
  Mode 09 (VIN, canned).
- CLI-selectable scenarios: `healthy` (default), `fault-injected` (raises
  P0171 + a freeze frame matching `docs/w203_reference.md`'s DTC table),
  `slow` (adds response latency near the P2max boundary), `unresponsive`
  (drops N frames, to exercise firmware's "re-init after 3 consecutive
  timeouts" rule from PLAN.md §5.1).
- `--port` / `--baud` flags for the real serial bench test; a `--fixture-dump`
  mode that emits canned request/response byte pairs as a C header, consumed
  by the native test suite, so native tests and the live simulator share one
  source of truth for "what does a fast-init response look like."

## 5. `kline_kwp`

Framework-agnostic (no `Arduino.h`, matching `pid_schema`'s pattern from
Phase 0) so it compiles under both `esp32dev` and `native`. Takes a
`KLineTransport&` in its constructor.

- Fast-init (0x81-style wake pattern) attempted first; 5-baud init fallback
  if fast-init doesn't get a valid response within its timeout — PLAN.md §5.1
  says both are required since it's `[Likely]` the SIM4LKE answers fast-init
  but unconfirmed until Phase 3.
- Tester-present keepalive sent on a timer during idle periods.
- Request/response framing: format byte + target + source + data + 8-bit
  sum-mod-256 checksum — this general KWP2000 structure and the checksum
  algorithm are `[Confirmed]` (ISO 14230-2). **Target address, exact
  timing constants (P2/P3/P4), and fast-init key bytes are `[Likely]`/
  `[Best estimate]`** — implemented as named constants with confidence-tag
  comments (e.g. `kTargetAddress = 0x33; // [Likely] placeholder — confirm
  against real SIM4LKE in Phase 3`), not buried magic numbers. `kline_sim.py`
  uses the *same* constants, so Phase 1's bench gate validates the protocol
  *logic* correctly even though the exact byte values may need correcting
  once real-car data exists.
- Mode 01/02/03/07/09 request builders + response parsers.
- Re-init after 3 consecutive request timeouts (PLAN.md §5.1).

## 6. `pid_scheduler`

Walks `pid_schema`'s Tier F/M/S entries at their target rates (~2 Hz / ~0.5 Hz
/ ~0.1 Hz) within the ~10 req/s K-line budget, calling into `kline_kwp` for
each poll and handing results to `ride_logger`. Tier `Legacy`/`Local` entries
are not scheduled here (Legacy: not actively polled per PLAN.md §2; Local
[`BATT_V_ADC`]: read directly off the ADC, no bus cost — stays out of this
module's scope for Phase 1, wired up when `power_mgr` lands).

## 7. `ride_logger`

Long-format CSV (`t_ms,signal,value`) via `RideStorage`. Header block: fw
version (`version.h`), which init mode succeeded (fast vs 5-baud), the
PID-availability bitmask observed during the bench run, DTC snapshot. Flush
periodically (mirrors the grad-project lesson in PLAN.md §2 about power-cut
resilience — even though Phase 1 has no real power-cut risk on a bench with
no car, the flush discipline is part of what the acceptance gate exercises).

## 8. Phase 1 acceptance (maps to PLAN.md's gate)

1. Native tests (no hardware): checksum/framing correctness, timeout →
   re-init behavior, tier-scheduling math, CSV formatting — all against
   fixtures sourced from `kline_sim.py --fixture-dump`.
2. Real bench run: `uv run firmware/tools/kline_sim.py --port <COMx> healthy`
   + firmware flashed to your ESP32 → completes init, polls the full
   schedule, produces a real ride CSV in LittleFS, pulled off and inspected.
3. A second real bench run with `--fault-injected` → CSV header block shows
   the injected DTC + freeze frame captured correctly.

## Self-review notes

- **Placeholder scan:** no TBD/TODO; every module has concrete responsibilities.
- **Internal consistency:** `KLineTransport`/`RideStorage` interfaces are
  referenced consistently across §1/§3/§5/§7; tier names (F/M/S/Local/Legacy)
  match `CLAUDE.md`'s schema table.
- **Scope:** one coherent spec — `kline_sim.py` and the firmware core are
  tightly coupled (the simulator exists to test the firmware), decomposing
  them into separate specs would just split one design in half.
- **Ambiguity:** the one deliberately-left-open area is exact KWP2000 byte
  values (target address, timing constants, fast-init key bytes) — this is
  intentional, not an oversight: those are `[Likely]`/`[Best estimate]` per
  PLAN.md §0.4 until Phase 3's real car probe, and the design isolates them
  as named, tagged constants specifically so that resolution doesn't require
  restructuring code later.
